# 기획 기본 구조

## 1. 프로젝트명
Echoes of Modernity: 근현대 미술, 방을 거닐다

## 2. 콘텐츠 개요
이 프로젝트는 근현대 미술의 변화 과정을 여러 전시 방을 직접 걸어 다니며 체험하는 PC 기반 1인칭 3D 가상 전시다. 플레이어는 각 공간에서 작품을 감상하고 `E` 키 상호작용을 통해 큐레이터 해설과 작품의 시대적 맥락을 확인한다. 미술사 지식이 많지 않은 사용자도 공간 분위기와 해설을 따라 전시의 흐름을 직관적으로 이해할 수 있도록 구성한다.

## 3. 핵심 컨셉
1. 방을 걸으며 이해하는 근현대 미술
- 사용자는 연표나 긴 설명 대신, 시대별로 분위기가 다른 전시 방을 직접 이동하며 미술의 변화 흐름을 체감한다.

2. 공간 분위기와 해설이 함께 만드는 직관적 감상
- 각 방은 조명, 색감, 작품 배치로 시대 감수성을 먼저 전달하고, 큐레이터 해설이 그 의미를 짧게 보강한다.

3. 단순한 상호작용으로 이어지는 몰입형 전시
- 플레이어는 `WASD` 이동, `Mouse` 시점, `E` 상호작용만으로 작품 해설, 심화 정보, 공간 장치를 자연스럽게 경험한다.

4. 프로토타입에 맞춘 명확한 전시 흐름
- 프롤로그 홀, 시대별 전시 방, 에필로그 라운지로 이어지는 구조를 통해 복잡한 분기 없이 시작-몰입-회고 흐름을 분명하게 전달한다.

## 4. 핵심 메커니즘
### 4-1. 플레이어 기본 탐색
- 목적: 플레이어가 방을 직접 걸으며 전시 흐름을 따라가게 한다.
- 핵심 구현 동작:
  - `WASD`로 이동한다.
  - `Mouse`로 시점을 회전한다.
  - 방과 방 사이를 단방향이 아닌 자유 이동으로 연결한다.
  - 각 방 입구에서 다음 감상 방향이 보이도록 동선을 단순하게 유지한다.
- 체크리스트:
  - [ ] 1인칭 이동과 시점 회전이 Old Input System 기준으로 동작한다.
  - [ ] 플레이어 이동 속도는 전시 감상용 보행 속도로 고정한다.
  - [ ] 각 방 입구에서 대표 작품 또는 입구 월 텍스트가 바로 보인다.

### 4-2. 작품 감상 상호작용
- 목적: 플레이어가 작품 앞에서 멈추고 해설을 여는 기본 감상 루프를 만든다.
- 핵심 구현 동작:
  - 작품 앞 상호작용 범위에 들어가면 `E` 안내 문구를 표시한다.
  - `E` 입력 시 작품 해설 패널을 연다.
  - 해설 종료 시 다시 자유 이동 상태로 복귀한다.
  - 한 번에 하나의 작품만 상호작용 가능하게 제한한다.
- 체크리스트:
  - [ ] 작품마다 상호작용 트리거가 배치되어 있다.
  - [ ] 상호작용 가능 상태에서만 `E` 안내 문구가 노출된다.
  - [ ] 해설 패널이 열린 동안 중복 입력 또는 중복 팝업이 발생하지 않는다.

### 4-3. 큐레이터 해설 전달
- 목적: 큐레이터를 공간 해석자 역할로 사용해 작품과 시대 맥락을 짧게 연결한다.
- 핵심 구현 동작:
  - 첫 `E` 입력 시 큐레이터의 1차 해설을 자막으로 재생한다.
  - 추가 입력 시 작품명, 작가명, 제작연도, 시대 키워드만 담은 심화 정보를 보여준다.
  - 해설 문장은 짧게 유지하고 행동 유도 문장을 포함한다.
  - 각 방 시작 지점에 큐레이터의 방 소개 해설 1개를 둔다.
- 체크리스트:
  - [ ] 작품별 1차 해설과 심화 정보가 분리되어 있다.
  - [ ] 심화 정보는 메타데이터 중심으로 최소 항목만 노출한다.
  - [ ] 각 방에 큐레이터 소개 해설 진입 포인트가 1개 이상 있다.

### 4-4. 방 진행과 전시 흐름 유지
- 목적: 플레이어가 길을 잃지 않고 프롤로그-전시 방-에필로그 순서의 흐름을 체감하게 한다.
- 핵심 구현 동작:
  - 프롤로그 홀에서 조작 안내와 전시 목적을 먼저 전달한다.
  - 각 방에는 대표 작품 1개와 방 소개 해설 1개를 핵심 지점으로 둔다.
  - 에필로그 라운지에서 관람 종료 메시지를 보여준다.
- 체크리스트:
  - [ ] 프롤로그에 기본 조작 안내와 첫 큐레이터 인사가 있다.
  - [ ] 각 방의 대표 감상 포인트가 명확히 구분된다.
  - [ ] 에필로그에서 관람 종료 메시지를 확인할 수 있다.

## 5. 기술 명세
### 5-1. 공통 시스템
- 구현 범위
  - 프로토타입은 `Exhibition` 단일 씬 기준으로 구성한다.
  - 입력은 Old Input System만 사용한다.
  - 새 C# 클래스명은 모두 `HO_` 접두사를 사용한다.
  - 클래스 파일명과 클래스명은 반드시 일치시킨다.
  - 클래스와 주요 함수에는 역할 설명 주석을 반드시 작성한다.
  - 구현 클래스는 `HO_GameManager`, `HO_UIManager`, `HO_ExhibitData`, `HO_PlayerController`, `HO_CuratorPresenter`, `HO_ExhibitionFlowController` 6개 이하로 유지한다.
- 클래스 구조
  - `HO_GameManager` / `MonoBehaviour` / 전시 시작, 현재 관람 상태, 공통 참조를 관리한다.
  - `HO_UIManager` / `MonoBehaviour` / 상호작용 안내, 해설 패널, 종료 메시지 UI를 제어한다.
  - `HO_ExhibitData` / `ScriptableObject` / 작품명, 해설, 심화 정보, 연결 리소스를 보관한다.
- 검증원칙
  - 전역 상태는 `HO_GameManager` 한 곳에서만 바뀐다.
  - 등장인물 기준 책임을 우선 묶고, 지금 한 클래스에서 해결 가능한 기능은 추가 클래스로 분리하지 않는다.
  - UI 열림 상태와 플레이어 조작 가능 상태가 항상 같이 검증된다.
  - 작품 데이터는 인스펙터에서 바로 확인 가능한 필드만 사용한다.
- 응답원칙
  - 입력이 없으면 상태를 바꾸지 않는다.
  - 잘못된 참조가 있으면 기능을 중단하고 로그로 원인을 남긴다.
  - 공통 UI는 한 번에 하나의 주요 패널만 연다.

### 5-2. 플레이어 탐색 시스템
- 구현 범위
  - `WASD` 이동과 `Mouse` 시점 회전만 구현한다.
  - 전시 감상용 보행 속도만 제공한다. 달리기와 점프는 제외한다.
  - 시작 위치는 프롤로그 홀 고정 지점으로 둔다.
  - 플레이어가 상호작용 대상 감지, `E` 입력 처리, 현재 감상 대상 선택까지 함께 맡는다.
- 클래스 구조
  - `HO_PlayerController` / `MonoBehaviour` / 이동, 회전, 중력, 조작 잠금, 전방 대상 감지, `E` 입력 기반 작품/공간 상호작용 요청을 한 클래스에서 처리한다.
- 검증원칙
  - 이동 입력은 Old Input System 축 입력으로만 읽는다.
  - UI가 열린 동안 이동과 회전이 모두 멈춰야 한다.
  - 상호작용 대상은 플레이어 기준 하나만 활성화되고, 대상이 바뀌면 이전 선택이 즉시 해제되어야 한다.
  - 벽 충돌 시 관통하지 않고 제자리에서 멈춰야 한다.
- 응답원칙
  - 이동 입력이 있으면 즉시 이동한다.
  - 시점 회전은 프레임마다 누적 적용한다.
  - `E` 입력은 현재 감지된 대상이 있을 때만 처리하고, 해설 단계 요청은 한 번씩만 전달한다.
  - 조작 잠금 상태에서는 입력을 읽어도 실제 이동에 반영하지 않는다.

### 5-3. 작품 상호작용 시스템
- 구현 범위
  - 작품 오브젝트는 별도 상호작용 전용 클래스를 두지 않고 콜라이더, 태그, `HO_ExhibitData` 연결로만 구성한다.
  - 작품 앞 트리거 진입, `E` 안내 노출, `E` 입력 상호작용은 `HO_PlayerController`가 처리한다.
  - 작품은 한 번에 하나만 감상 대상으로 유지한다.
- 클래스 구조
  - `HO_PlayerController` / `MonoBehaviour` / 트리거 진입/이탈 감지, 현재 작품 선택, `HO_CuratorPresenter` 호출을 처리한다.
  - `HO_ExhibitData` / `ScriptableObject` / 작품 해설 단계에 필요한 데이터만 제공한다.
- 검증원칙
  - 상호작용 가능한 작품이 없을 때는 안내 UI가 보이지 않아야 한다.
  - 같은 작품에서 `E`를 연속 입력해도 해설 단계가 중복 실행되지 않아야 한다.
  - 플레이어가 트리거를 벗어나면 현재 대상과 안내 UI가 즉시 정리되어야 한다.
- 응답원칙
  - 대상 진입 시 `HO_UIManager`를 통해 안내 UI를 켠다.
  - `E` 입력 시 `HO_PlayerController`가 현재 상태에 맞는 해설 단계만 한 번 요청한다.
  - 대상 이탈 또는 닫기 시 `HO_PlayerController`가 안내 UI와 선택 상태를 정리한다.

### 5-4. 큐레이터 해설 시스템
- 구현 범위
  - 방 소개 해설, 작품 1차 해설, 심화 정보 2단계만 제공한다.
  - 자막은 측면 패널 또는 하단 텍스트 박스 한 종류로 통일한다.
  - 텍스트만으로 진행하게 만든다.
  - 큐레이터 연출은 해설 표시와 함께 한 클래스에서 해결하고, 별도 연출 제어 클래스를 추가하지 않는다.
  - 설명을 위해 플레이어와 2미터 떨어진 위치로 이동한다.

- 클래스 구조
  - `HO_CuratorPresenter` / `MonoBehaviour` / 해설 단계 전환, 자막 문장 표시, 심화 정보 노출, 큐레이터 등장 위치/연출 갱신을 한 클래스에서 처리한다.

- 검증원칙
  - 1차 해설과 심화 정보가 같은 필드에 섞이지 않아야 한다.
  - 해설 대상이 바뀌면 이전 문장이 남지 않아야 한다.
  - 빈 해설 데이터는 패널을 열지 않고 로그만 남긴다.
  - 플레이어와 2미터 떨어진 위치로 이동해야 한다.
- 응답원칙
  - 플레이어와 2미터 떨어진 위치로 도착한 후 해설이 시작된다.
  - 첫 호출에는 짧은 해설을 보여준다.
  - 다음 호출에는 작품 메타데이터 중심의 심화 정보를 보여준다.
  - 종료 호출에는 해설 UI를 닫고 플레이어 제어를 돌려준다.

### 5-5. 전시 흐름 시스템
- 구현 범위
  - 프롤로그 진입, 방 소개 포인트, 에필로그 종료 메시지만 상태로 관리한다.
  - 방 완료 판정은 대표 작품 감상 여부 또는 지정 트리거 진입 여부 중 하나로 단순화한다.
  - 분기 진행과 저장 기능은 제외한다.
  - 방 단위 흐름 제어만 맡고, 작품 단위 감상 로직은 플레이어/큐레이터 클래스에 넘긴다.
- 클래스 구조
  - `HO_ExhibitionFlowController` / `MonoBehaviour` / 현재 방 상태, 대표 포인트 통과, 프롤로그/방 소개/에필로그 트리거 판정, 다음 방 안내를 한 클래스에서 관리한다.
- 검증원칙
  - 방 상태는 역순으로 건너뛰지 않아야 한다.
  - 같은 구간 트리거에 재진입해도 완료 처리가 중복되지 않아야 한다.
  - 에필로그 진입 시 종료 메시지가 한 번만 출력되어야 한다.
- 응답원칙
  - `HO_ExhibitionFlowController`가 지정 구간 트리거 진입 시 현재 진행 상태를 갱신한다.
  - 아직 보지 않은 대표 포인트가 있으면 다음 안내를 유지한다.
  - 종료 상태가 되면 추가 안내보다 관람 종료 메시지를 우선 표시한다.

### 5-6. 프로토타입 공통 리소스
- 구현 범위
  - 작품 메타데이터, UI 프리팹, 트리거 규칙, 최소 테스트 항목만 정의한다.
  - 리소스 구조는 인스펙터 연결이 쉬운 수준으로 유지한다.
  - 작품 오브젝트는 데이터 연결과 콜라이더 설정 중심으로 두고, 작품마다 별도 스크립트를 추가하지 않는다.
- 클래스 구조
  - `HO_ExhibitData` / `ScriptableObject` / `작품 ID`, `작품명`, `작가명`, `제작연도`, `매체`, `대표 키워드`, `1차 해설`, `심화 정보`를 보관한다.
- 검증원칙
  - 작품마다 `HO_ExhibitData`가 1개씩 연결되어야 한다.
  - 상호작용 트리거 오브젝트는 작품 오브젝트와 분리하거나 자식으로 두되, 별도 상호작용 클래스는 붙이지 않는다.
  - UI 프리팹은 안내 프롬프트, 해설 패널, 종료 메시지 세 종류만 우선 사용한다.
- 응답원칙
  - 리소스 누락 시 대체 동작보다 오류 로그를 우선 남긴다.
  - 테스트는 `이동`, `시점 회전`, `E 상호작용`, `해설 2단계`, `종료 메시지` 순서로 확인한다.
